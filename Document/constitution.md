## 1. 宪法的地位与解释

### 1.1 效力层级

本项目文档与代码的效力层级如下（由高到低）：

```
constitution.md (开发宪法) —— 最高约束，变更需经项目 owner 批准
    ↑
spec.md (产品需求规格) —— 需求基线，所有设计必须追溯至此
    ↑
plan-*.md (模块设计计划) —— 模块级设计与测试用例，编码的唯一依据
    ↑
tasks.md (实现任务集合) —— 由 plan 拆分的可执行任务清单
    ↑
source code (源代码) —— 设计的实现，必须与 plan 保持一致
```

### 1.2 解释原则

| 原则 | 说明 |
|------|------|
| **上层优先** | 当文档之间出现冲突时，效力层级高的文档为准 |
| **显式优于隐式** | 任何未在设计文档中明确记录的实现细节，必须补充文档后才能进入代码 |
| **可追溯性** | 每一行代码必须能追溯到至少一个设计决策，每一个设计决策必须能追溯到至少一个需求项 |

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| **plan** | 模块设计计划文档，包含接口定义、数据流、测试用例。编码前的强制性设计产出物 |
| **设计基线** | 经评审锁定、进入编码阶段的计划文档版本 |
| **需求追溯链** | `需求项 (spec) → 设计决策 (plan) → 实现任务 (tasks) → 代码实现 (source)` 的完整链条 |
| **质量门禁** | 代码合入前必须通过的检查项集合，未通过则禁止合入 |

## 2. 目录结构约定

### 2.1 顶层目录划分

```
gis-agent/
├── Document/               # 文档、资源、设计产出物
│   ├── constitution.md     # 开发宪法（本文件）
│   ├── spec.md             # 产品需求规格说明书（PRD）
│   ├── plan-*.md           # 模块设计计划（按模块），含设计 + 测试用例
│   ├── ADR-*.md            # 架构决策记录（Architecture Decision Records）
│   ├── assets/             # 文档引用的图片、图表等资源
│   └── references/         # 外部参考资料（HTML 原始文档等），不纳入 Git
│
├── SourceCode/             # 源码、开发资源
│   ├── src/                # 源代码目录
│   │   ├── core/           # 核心领域逻辑（模板引擎、安全层、工作空间）
│   │   ├── rag/            # RAG 管道（文档加载、切分、嵌入、检索）
│   │   ├── llm/            # LLM 交互层（意图识别、参数抽取、问答）
│   │   ├── cli/            # 命令行界面与交互状态机
│   │   └── templates/      # Jinja2 模板文件（*.j2）
│   ├── tests/              # 测试代码
│   │   ├── unit/           # 单元测试
│   │   ├── integration/    # 集成测试
│   │   └── fixtures/       # 测试数据与 mock 文件
│   ├── scripts/            # 开发辅助脚本（初始化、构建等）
│   ├── docs/               # 源码级文档（README、API 文档）
│   ├── tasks/              # 实现任务集合（由 plan 拆分的可执行任务清单）
│   └── pyproject.toml      # 项目配置与依赖声明
│
└── .gitignore
```

### 2.2 目录边界规则

| 规则编号 | 规则内容 | 违规处理 |
|:--------:|---------|---------|
| DIR-1 | **Document 下绝不放置可执行代码**。设计文档、架构图、参考资料仅限于此目录。 | 代码文件直接退回，要求迁移至 SourceCode |
| DIR-2 | **SourceCode 下绝不放置设计文档**。README 除外，但 README 仅用于说明如何构建/运行，不承载设计决策。 | 设计文档直接退回，要求迁移至 Document |
| DIR-3 | **所有 plan 必须位于 Document/ 根目录**，命名格式为 `plan-{模块简称}.md`，如 `plan-rag.md`、`plan-security.md`。 | 不符合命名规范的 plan 视为无效 |
| DIR-4 | **所有 ADR 必须位于 Document/ 根目录**，命名格式为 `ADR-{NNNN}-{决策简称}.md`，如 `ADR-0001-use-chromadb.md`。 | 不符合命名规范的 ADR 视为无效 |
| DIR-5 | **Jinja2 模板文件 (*.j2) 必须位于 SourceCode/src/templates/**，按功能子目录组织。 | 散落的模板文件必须归集 |

## 3. 规范驱动设计工作流

### 3.1 工作流总览

本项目严格执行**设计先行、文档驱动、编码后置**的工作流。任何代码提交必须有前置的设计文档支撑。

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  需求分析    │ →  │  架构设计    │ →  │  模块设计    │ →  │  任务细化    │ →  │   编码实现   │
│  (spec.md)  │     │ (plan-Arch) │     │  (plan-XX)  │     │  (tasks.md) │     │  (source)   │
└─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
       ↑                                                                                  │
       └────────────────────────── 验收与追溯 ←────────────────────────────────────────────┘
```

### 3.2 阶段定义与产出物

| 阶段 | 准入条件 | 产出物 | 评审要求 |
|------|---------|--------|---------|
| **P0: 需求分析** | 用户/项目 owner 提出需求 | `spec.md` 更新 | 与项目 owner 确认 |
| **P1: 架构设计** | spec.md 中对应需求项已锁定 | `plan-architecture.md` | 必须进行架构评审 |
| **P2: 模块设计** | 架构设计已锁定 | `plan-{模块}.md` | 模块 owner 自审 + 交叉评审 |
| **P2.5: 任务细化** | 模块设计已锁定 | `tasks.md` 更新 | 将 plan 拆解为可执行任务 |
| **P3: 编码实现** | 任务清单已就绪 | 源代码 + 单元测试（TDD 驱动） | 代码审查 (Code Review) |
| **P4: 集成验证** | 单元测试全部通过 | 集成测试报告 | 与验收标准 (AC) 逐项核对 |

### 3.3 阶段转换门禁

阶段转换必须满足以下条件，**缺一不可**：

**P0 → P1（需求 → 架构）**
- [ ] spec.md 中对应需求项已明确编号且未标记为 "Won't"
- [ ] 需求项已分配到具体的开发迭代

**P1 → P2（架构 → 模块）**
- [ ] plan-architecture.md 已创建并包含模块划分图
- [ ] 每个模块的接口契约已初步定义
- [ ] 架构评审会议已召开并通过

**P2 → P2.5（模块设计 → 任务细化）**
- [ ] plan-{模块}.md 已完成且包含：模块职责、接口定义、数据流图、异常处理策略、测试用例
- [ ] 设计文档中的每个设计决策都有需求追溯（引用 spec.md 中的需求 ID）
- [ ] 模块 owner 自审通过
- [ ] 至少一名其他开发者交叉评审通过
- [ ] tasks.md 已更新，包含本模块的可执行任务清单

**P2.5 → P3（任务细化 → 编码）**
- [ ] tasks.md 中对应任务已拆解到可编码粒度
- [ ] 每个任务关联到 plan 中的具体设计决策
- [ ] plan 中的每个测试用例已转化为可执行的**失败测试骨架**（failing test），且测试文件已提交
- [ ] **不允许先实现代码后补测试**。进入编码阶段的第一笔代码必须是测试代码

**P3 → P4（编码 → 集成）**
- [ ] 单元测试覆盖率 ≥ 80%
- [ ] **所有 plan 中定义的测试用例均有对应的通过测试**，且测试先于实现代码提交
- [ ] 代码通过静态检查（ruff + mypy）
- [ ] 代码审查 (Code Review) 通过，Review 意见已全部解决
- [ ] 代码中的每个公共函数/类都有 docstring，且与 plan 中的接口定义一致

### 3.4 禁止行为（红色条款）

| 编号 | 禁止行为 | 违规后果 |
|:----:|---------|---------|
| RED-1 | **无 plan 直接编码**。任何进入 git 历史的功能代码，必须有对应的 plan 文档先行合入。 | 代码提交被拒绝，强制回退 |
| RED-2 | **plan 与代码不一致**。设计文档中定义的接口、行为与代码实现出现偏差。 | 视为缺陷，必须修复文档或代码使之一致 |
| RED-3 | **未经评审的设计直接进入编码**。plan 未通过评审即开始实现。 | 已编码部分视为无效，需重新评审 |
| RED-4 | **需求变更不更新 spec**。需求发生变更但未同步更新 spec.md。 | 后续基于旧需求的 plan 全部失效，需重新设计 |
| RED-5 | **直接修改已锁定的 plan**。已进入编码阶段的设计文档未经变更控制流程直接修改。 | 修改无效，代码基线可能被迫回退 |

## 4. 设计文档规范

### 4.1 plan 标准结构

每个 plan 必须包含以下章节（不可省略，无内容则标注 "N/A"）：

```markdown
# plan-{模块名}

## 1. 设计概述
- 模块职责（一句话描述）
- 所属架构层次
- 对应需求项（引用 spec.md 中的 ID）

## 2. 设计决策
- 每个决策的编号 (DC-{NNNN})、内容、理由、替代方案对比
- 引用对应的 ADR（如有）

## 3. 接口定义
- 公共 API / 函数签名（含类型注解）
- 输入/输出数据结构（含字段类型、约束、默认值）
- 异常抛出约定

## 4. 数据流与控制流
- 序列图或流程图（文字描述也可，但需清晰）
- 状态机定义（如涉及）

## 5. 依赖关系
- 向上依赖（本模块依赖的其他模块）
- 向下暴露（其他模块可使用的公共接口）

## 6. 异常与错误处理
- 错误码 / 异常类型定义
- 每种错误的处理策略（重试、降级、报错）

## 7. 测试策略
- 单元测试覆盖范围
- 集成测试场景
- Mock 策略

## 8. 需求追溯表
| 需求 ID | 设计决策 | 代码文件/函数 |
|---------|---------|--------------|
```

### 4.2 ADR 标准结构

每个架构决策记录必须包含以下章节：

```markdown
# ADR-{NNNN}: {决策标题}

## 状态
- 提议 / 已通过 / 已否决 / 已废弃

## 上下文
- 需要做出决策的问题背景

## 决策
- 明确的选择

## 后果
- 正面影响
- 负面影响 / 风险
- 迁移成本（如替换此决策）

## 替代方案
- 被否决的方案及否决理由
```

### 4.3 追溯性要求

| 追溯方向 | 要求 | 实现方式 |
|---------|------|---------|
| 需求 → 设计 | 每个需求项 (F*) 必须有至少一个 plan 设计决策覆盖 | plan 中 "对应需求项" 章节显式引用 |
| 设计 → 任务 | 每个设计决策 (DC-*) 必须在 tasks.md 中有对应任务 | tasks 中引用 DC 编号 |
| 任务 → 代码 | 每个任务必须在代码中有对应实现 | 代码注释中引用 task ID |
| 代码 → 测试 | 每个公共函数必须有至少一个单元测试 | 测试文件名与模块对应，测试函数名与被测函数对应 |

## 5. 编码规范

### 5.1 语言与风格

| 属性 | 规范 |
|------|------|
| 语言版本 | Python 3.10+ |
| 类型注解 | **强制要求**。所有函数参数和返回值必须有类型注解，使用 `typing` 模块 |
| 代码格式化 | `ruff format` 为标准 |
| 代码检查 | `ruff check` + `mypy --strict` 为强制门禁 |
| 行长度 | 88 字符（Black 默认） |
| 编码 | UTF-8 |

### 5.2 项目特定规范

| 规则编号 | 规范内容 | 说明 |
|:--------:|---------|------|
| CODE-1 | **所有 GDAL 命令必须通过 Jinja2 模板渲染生成** | 严禁在代码中拼接 GDAL CLI 字符串，参见 spec.md P1 |
| CODE-2 | **所有文件路径操作必须经过规范化层** | 调用 `workspace.resolve_path()` 或等效函数进行路径规范化，禁止直接使用 `os.path.join` 处理用户输入路径 |
| CODE-3 | **所有 LLM 调用必须封装在 `llm/` 模块内** | 其他模块禁止直接调用 `anthropic` 客户端 |
| CODE-4 | **所有 ChromaDB 操作必须封装在 `rag/` 模块内** | 其他模块禁止直接操作向量数据库 |
| CODE-5 | **异常不得静默吞没** | 所有 `except` 块必须至少记录日志或向上转换异常 |
| CODE-6 | **模板文件 (*.j2) 中的参数必须做转义处理** | 防止命令注入，参见 plan-security.md 中的转义策略 |

### 5.3 文档字符串规范

```python
def function_name(param: str, optional: int = 0) -> bool:
    """简短描述（一行）。

    详细描述（如有必要）。说明函数做什么、不做什么。

    Args:
        param: 参数说明。
        optional: 参数说明，含默认值含义。

    Returns:
        返回值说明。

    Raises:
        ValueError: 何时抛出。
        FileNotFoundError: 何时抛出。

    Design:
        对应设计决策 DC-XXXX。
    """
```

### 5.4 TDD 测试驱动开发

本项目编码阶段严格执行 **TDD（Test-Driven Development）**。

#### 5.4.1 红-绿-重构循环

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   红 (RED)   │ →  │  绿 (GREEN)  │ →  │  重构(REFACTOR)│
│ 写一个失败的  │     │ 写最小量代码  │     │ 优化代码结构  │
│   测试       │     │  使测试通过   │     │ 不改变行为    │
└─────────────┘     └─────────────┘     └─────────────┘
       ↑                                            │
       └──────────── 下一轮循环 ←───────────────────┘
```

每个公共函数/方法的诞生必须经历完整的三步循环：

1. **红**：根据 plan 中的测试用例，编写一个**确定会失败的测试**。测试代码先于实现代码进入版本控制。
2. **绿**：编写**最小量**的实现代码使当前测试通过。允许在此阶段写 "脏代码"，只要测试通过即可。
3. **重构**：在测试绿灯保护下，优化代码结构（消除重复、改善命名、拆分函数）。每一步重构后必须确认所有测试仍通过。

#### 5.4.2 TDD 与规范驱动设计的衔接

| 规范驱动阶段 | TDD 对应动作 |
|-------------|-------------|
| P2 模块设计 | plan 中输出测试用例（输入/输出、边界条件、异常场景） |
| P2.5 任务细化 | 将 plan 中的测试用例拆分为具体测试函数清单 |
| **P3 编码：第一步** | **提交 failing test（红），无实现代码** |
| P3 编码：第二步 | 提交使测试通过的最小实现（绿） |
| P3 编码：第三步 | 提交重构后的代码（重构），保持测试通过 |

#### 5.4.3 TDD 纪律（不可违反）

| 纪律编号 | 内容 | 违反后果 |
|:--------:|------|---------|
| TDD-1 | **测试先于实现**。任何新增公共接口，其测试代码必须在实现代码之前提交到 git。 | Code Review 不通过，要求拆分提交 |
| TDD-2 | **一个失败测试只写一个**。不允许一次性写多个 failing test 再批量实现。 | 视为偏离 TDD，Review 时必须指出 |
| TDD-3 | **测试必须可独立运行**。每个测试函数不依赖其他测试的执行顺序或副作用。 | 视为缺陷，必须修复 |
| TDD-4 | **重构必须有测试保护**。任何重构提交时，相关测试必须全部通过。 | 重构被视为无效，需回退 |
| TDD-5 | **plan 中的测试用例必须全部覆盖**。plan 中列出的每个测试场景，代码中必须有对应测试。 | 视为测试遗漏，P3→P4 门禁不通过 |

## 6. 架构设计原则

### 6.1 分层架构

系统采用严格的分层架构，上层可调用下层，**严禁反向依赖**。

```
┌─────────────────────────────────────┐
│  CLI 层 (cli/)                      │  用户交互、状态机、命令解析
├─────────────────────────────────────┤
│  核心层 (core/)                     │  工作空间管理、模板引擎、安全校验
├─────────────────────────────────────┤
│  应用层 (llm/ + rag/)               │  LLM 交互、意图识别、文档检索
├─────────────────────────────────────┤
│  基础设施层 (外部依赖)               │  anthropic、chromadb、jinja2、GDAL CLI
└─────────────────────────────────────┘
```

### 6.2 依赖规则

| 规则 | 内容 |
|------|------|
| DEP-1 | CLI 层可依赖 core、llm、rag 层 |
| DEP-2 | core 层可依赖 llm、rag 层（用于模板渲染和参数校验） |
| DEP-3 | llm 层仅可依赖 rag 层（用于 RAG 增强问答） |
| DEP-4 | rag 层不依赖任何上层模块，但可依赖 config/（配置基础设施） |
| DEP-5 | 任何层都可通过接口抽象依赖外部库，但禁止直接暴露外部库类型到上层 |

### 6.3 设计模式约定

| 场景 | 推荐模式 | 禁止模式 |
|------|---------|---------|
| 模板渲染 | Jinja2 模板引擎 | 字符串拼接 |
| 意图分类 | 策略模式 + 模板注册表 | 大段 if/else 链 |
| 工作空间管理 | 单例模式（进程级） | 全局变量直接传递路径 |
| LLM 调用 | 适配器模式（封装 anthropic 客户端） | 直接散布 API 调用 |
| 参数校验 | 校验器链（Chain of Responsibility） | 单个巨型校验函数 |

## 7. 变更控制

### 7.1 变更类型与流程

| 变更类型 | 触发条件 | 处理流程 |
|---------|---------|---------|
| **需求变更** | spec.md 内容发生增删改 | 1. 在 spec.md 中标记变更并更新版本<br>2. 评估影响的 plan 范围<br>3. 按影响程度决定：更新 plan / 重新设计 / 拒绝变更 |
| **设计变更** | plan 中接口/行为发生变更 | 1. 检查变更原因（是需求变更还是设计缺陷）<br>2. 更新 plan 并记录变更日志<br>3. 评估对 tasks.md 和已实现代码的影响<br>4. 更新或回退受影响代码 |
| **代码缺陷** | 实现与 plan 不符 | 1. 判断是代码错误还是设计缺陷<br>2. 代码错误 → 修复代码<br>3. 设计缺陷 → 按设计变更流程处理 |

### 7.2 变更影响评估矩阵

| 变更层级 | 影响范围 | 是否需要重新评审 |
|---------|---------|---------------|
| spec.md 需求项增删 | 所有关联 plan + 代码 | 是，全量架构评审 |
| plan 接口签名变更 | 本模块 + 依赖本模块的上层 + tasks.md | 是，模块交叉评审 |
| plan 内部实现调整（接口不变） | 仅本模块代码 + 对应 tasks | 否，模块 owner 自审即可 |
| 代码 Bug 修复（行为不变） | 仅本模块代码 | 否，Code Review 即可 |

### 7.3 版本管理

| 文档类型 | 版本格式 | 升级时机 |
|---------|---------|---------|
| constitution.md | v{Major}.{Minor}.{Patch} | Major: 宪法条款变更；Minor: 流程优化；Patch: 文字勘误 |
| spec.md | v{Major}.{Minor} | Major: 需求范围变更；Minor: 需求细节补充 |
| plan-*.md | v{Major}.{Minor} | Major: 接口/架构变更；Minor: 实现细节补充 |
| ADR-*.md | 无版本，以状态标记 | 新建/通过/废弃 |

## 8. 质量门禁

### 8.1 提交前检查（Pre-Commit）

每次代码提交前必须运行并通过以下检查：

```bash
# 代码格式化
ruff format --check src/ tests/

# 代码风格与错误检查
ruff check src/ tests/

# 类型检查
mypy --strict src/

# 单元测试
pytest tests/unit/ --cov=src --cov-report=term-missing --cov-fail-under=80
```

### 8.2 代码审查（Code Review）检查单

审查人必须逐项确认：

- [ ] **追溯性**：代码中的每个公共接口在 plan 中有定义，且引用了正确的设计决策编号
- [ ] **一致性**：代码行为与 plan 描述一致
- [ ] **安全**：无直接路径拼接、无命令注入风险、无静默异常吞没
- [ ] **测试**：新增代码有对应的单元测试，测试覆盖边界情况
- [ ] **规范**：符合编码规范（类型注解、docstring、命名约定）
- [ ] **依赖**：未引入未经批准的第三方库

### 8.3 集成门禁

合入主干（main）前必须通过：

- [ ] 所有单元测试通过
- [ ] 所有集成测试通过（如有）
- [ ] 覆盖率 ≥ 80%
- [ ] ruff + mypy 零报错
- [ ] Code Review 至少一人通过
- [ ] 对应的 plan 已锁定且版本与代码一致

## 9. 安全与合规

### 9.1 安全原则（源自 spec.md）

本项目所有开发活动必须严格遵守 spec.md 中定义的五大设计原则：

| 原则编号 | 原则名称 | 编码层面的体现 |
|:----:|---------|--------------|
| P1 | **模板化命令，杜绝幻觉** | 所有 GDAL 命令字符串必须通过 `src/templates/` 下的 Jinja2 模板渲染，严禁任何动态字符串拼接生成命令 |
| P2 | **先展后行** | CLI 层必须在调用执行层前打印完整脚本内容，并获得用户明确确认（Y/N） |
| P3 | **最小权限** | 默认输出到工作空间内，输出文件加时间戳防覆盖；所有路径经规范化后使用 |
| P4 | **文档知识只读** | RAG 模块仅读取 `SourceCode/data/` 下的本地文档 chunks，禁止调用外部 API 获取知识 |
| P5 | **依赖极简** | `pyproject.toml` 的生产依赖以 `anthropic`、`chromadb`、`jinja2` 为主；`sentence-transformers` 作为 embedding 模型加载的必要依赖，允许纳入生产依赖 |

### 9.2 安全编码红线

| 编号 | 红线内容 | 违反后果 |
|:----:|---------|---------|
| SEC-1 | 禁止在代码中硬编码 API Key、密码等凭证 | 视为严重安全事件，提交被拒绝 |
| SEC-2 | 禁止执行用户输入的任何字符串作为命令 | 视为严重安全事件，提交被拒绝 |
| SEC-3 | 禁止在日志中输出敏感信息（API Key、完整文件内容） | 视为缺陷，必须修复 |
| SEC-4 | 所有用户输入的路径必须经过规范化（`os.path.normpath`）和校验后方可使用 | 不符合视为缺陷 |
| SEC-5 | 模板渲染后的命令字符串必须经安全校验层二次检查后方可提交执行 | 不符合视为缺陷 |
