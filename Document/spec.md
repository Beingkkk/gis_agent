## 1. 项目概述

### 1.1 目标

为内部 GIS 团队提供一个**命令行助手**，具备以下核心能力：

| 能力 | 说明 |
|------|------|
| 文档问答 | 基于本地 GDAL 帮助文档，解答工具使用问题 |
| 任务脚本化 | 接受自然语言数据处理需求，生成可审查的批处理脚本 |
| 安全执行 | 用户确认后在指定工作空间内执行脚本 |

### 1.2 定位

- **维护模式**：单人维护
- **使用范围**：内部工具，不对外发布
- **设计哲学**：简单可靠、零外部服务依赖

---

## 2. 用户与场景

### 2.1 目标用户

团队内的 **GIS 分析师** 与 **数据工程师**，具备基础空间数据概念知识。

### 2.2 典型使用场景

| 场景编号 | 用户输入示例 |
|----------|-------------|
| S1 | "把这三个省的 shp 合并成一个 GeoJSON，坐标系转 4326" |
| S2 | "用这个边界 shp 裁剪那批 tif" |
| S3 | "ogr2ogr 里怎么忽略无效字段？" |

## 3. 功能需求（MoSCoW）

### 3.1 Must — 必须实现

以下功能构成系统的最小可用产品（MVP），必须在首期交付。

| ID | 功能模块 | 需求描述 | 关键设计点 |
|:---:|---------|---------|-----------|
| **F1** | **文档问答** | 基于 J2 模板元数据（`@concept`、`@note`、`@common_error` 等）和 LLM 参数知识回答用户问题。基础概念由 LLM 直接回答；用法指导基于模板元数据；错误诊断结合模板 `@common_error` 与 LLM 推理。 | 模板注释格式扩展；Q&A 分层策略；模板匹配逻辑 |
| **F2** | **意图→模板映射** | 将用户自然语言分类到预定义的工作流模板（如 `shp2geojson`、`clip_raster`）。 | 模板注册表；意图分类 Prompt 设计；兜底策略 |
| **F3** | **参数抽取与追问** | 从输入中提取文件路径、坐标参考、选项等参数；缺失时主动向用户追问。 | 参数 Schema 定义；必填/可选校验规则；追问话术模板 |
| **F4** | **脚本生成** | 根据模板 + 参数，使用 Jinja2 渲染生成可执行的批处理脚本（`.bat`/`.sh`）。 | 模板语法规范；跨平台路径处理；参数转义安全 |
| **F5** | **用户确认与执行** | 向用户完整展示脚本内容，要求明确确认（Y/N）后，在指定工作空间内执行。 | 确认交互流程；执行超时控制；进程隔离 |
| **F6** | **安全与隔离** | 默认操作目录为工作空间，支持外部数据路径；禁止静默覆盖（输出文件默认加时间戳）；支持 `--dry-run` 空跑模式。 | 路径规范化；文件名时间戳规则；空跑模式实现 |
| **F7** | **工作空间管理** | 支持启动时指定工作空间（`--workspace`）作为默认 cwd 和 Agents.md 探测位置；数据路径不限定在该目录内。 | CLI 参数解析；工作空间路径规范化；目录存在性校验 |
| **F8** | **会话内记忆** | 单次对话中保留上下文（多轮追问和补充），避免信息丢失。 | 消息列表结构；Token 预算管理；上下文截断策略 |

### 3.2 Should — 应该实现

以下功能在 MVP 完成后优先实现，提升用户体验。

| ID | 功能模块 | 需求描述 | 依赖 |
|:---:|---------|---------|------|
| **F9** | **多步任务** | 支持简单的链式操作，如"裁剪 → 重投影 → 转格式"。 | F2、F4 |
| **F10** | **错误诊断** | 执行失败时，捕获 GDAL 错误信息并给出简单修复建议。 | F5、F1 |
| **F11** | **任务入口手册** | 自动读取工作空间下的 `Agents.md`，作为项目背景、默认参数等长期记忆注入系统提示词。 | F8 |

### 3.3 Could — 可以扩展

以下功能作为未来可选增强项，当前阶段仅预留接口。

| ID | 功能模块 | 需求描述 | 备注 |
|:---:|---------|---------|------|
| **F12** | **处理历史** | 记录每次执行的脚本和日志到工作空间内，便于回溯。 | 需设计日志目录结构与轮转策略 |
| **F13** | **可视化预览** | 生成前后数据的基本统计或快速地图截图（需额外轻量工具）。 | 强依赖外部可视化库，暂不纳入核心 |

### 3.4 Won't — 暂不实现

以下功能明确排除在当前版本范围外：

- 复杂空间分析（网络分析、空间统计）
- 多用户权限管理
- GUI 界面

## 4. 技术约束与选型

### 4.1 依赖栈

| 层级 | 技术项 | 版本/说明 |
|------|--------|----------|
| 运行环境 | Python | 3.10+ |
| GDAL 环境 | Conda + conda-forge | `conda install -c conda-forge gdal` |
| GDAL CLI 工具 | `ogr2ogr`、`gdalwarp`、`gdal_translate` 等 | 随 Conda 包提供 |
| LLM API | Claude 兼容 API | 支持切换本地模型 |
| 模板引擎 | **Jinja2** | 脚本渲染与知识元数据载体 |
| 核心依赖包 | `anthropic`、`jinja2` | 仅两项第三方库（P5 约束） |

### 4.2 LLM 使用边界

| 允许使用 LLM 的场景 | 禁止使用 LLM 的场景 |
|-------------------|-------------------|
| 意图识别与分类 | 直接生成 GDAL 命令字符串 |
| 参数抽取与追问 | 直接执行文件系统操作 |
| 文档问答（基于模板元数据） | 脱离模板的自由脚本生成 |

### 4.3 架构约束

- **会话状态**：单次对话内保留上下文（内存列表），退出即丢弃
- **持久化策略**：初期不持久化会话历史
- **依赖最小化**：新增依赖仅限 Jinja2，要求轻量、成熟、无复杂配置

### 4.4 模板知识源可靠性说明

本节说明系统如何通过 J2 模板元数据确保文档问答（F1）的回答基于**结构化、可执行、人工验证**的知识，而非自由检索或 LLM 幻觉。

#### 4.4.1 知识双层结构

| 层级 | 来源 | 用途 | 准确性保障 |
|------|------|------|-----------|
| **基础概念** | LLM 参数知识 | 回答"TIF 是什么""EPSG 是什么"等概念性问题 | LLM 训练数据中的通用知识 |
| **用法指导** | J2 模板元数据（`@concept`、`@note`、`@param`） | 回答"怎么把 SHP 转成 GeoJSON""参数怎么填" | 人工编码的可执行模板，与系统行动能力同源 |
| **错误诊断** | `@common_error` + LLM 推理 | 分析执行失败原因并给出修复建议 | 模板中记录的典型错误模式 |

#### 4.4.2 模板作为知识卡片的机制

每个 `.j2` 模板同时承载"可执行脚本"和"结构化知识"双重角色：

```
模板文件: vector/shp2geojson.j2
    │
    ├──→ 执行角色: 渲染为 ogr2ogr 命令脚本
    │
    └──→ 知识角色:
            ├── @description —— 功能描述（一句话）
            ├── @concept "GeoJSON" —— 格式概念解释
            ├── @param input file_path —— 参数定义与约束
            ├── @note "输出路径自动加时间戳" —— 使用提示
            └── @common_error "Unable to open datasource" —— 常见错误
```

#### 4.4.3 约束保证

| 约束层级 | 机制 | 效果 |
|---------|------|------|
| **数据源约束** | 模板元数据与可执行脚本同源，人工编写并纳入 Git 版本控制 | 知识质量可控，可追溯 |
| **Prompt 约束** | 用法指导类问题的系统 Prompt 明确 instruct："仅根据以下模板元数据回答，不要编造模板中未定义的参数" | 禁止 LLM 凭记忆编造命令用法 |
| **行动一致性** | Q&A 回答的知识与后续脚本生成使用同一套模板 | 消除"说得对但做不了"的脱节 |

#### 4.4.4 潜在偏差与缓解

| 风险 | 原因 | 缓解措施 |
|------|------|---------|
| 模板覆盖不全 | 新增 GDAL 功能暂无对应模板 | 开发新模板时以 GDAL HTML 文档为参考，将关键知识编码进元数据 |
| LLM 概念幻觉 | 基础概念回答依赖 LLM 参数知识，有训练截止日期 | GDAL 核心概念变化极慢；对版本特定细节引导用户到模板用法 |
| 模板元数据缺失 | 老模板未补充 `@concept`/`@common_error` | 渐进式补充，作为模板维护的一部分 |

## 5. 工作空间与记忆机制

### 5.1 工作空间（Workspace）

#### 定义

一个文件系统目录，作为所有操作的根路径。

#### 设置方式

| 方式 | 优先级 | 示例 |
|------|--------|------|
| 启动参数 `--workspace` | 高 | `python main.py --workspace /path/to/project` |
| 默认回退 | 低 | 当前工作目录（`os.getcwd()`） |

#### 约束规则

| 规则编号 | 规则描述 | 校验时机 |
|:--------:|---------|---------|
| W1 | 所有生成脚本和输出文件默认放置在工作空间下 | 脚本生成时 |
| W2 | 脚本执行时，将工作空间作为进程的当前目录（`cwd`） | 脚本执行时 |
| W3 | 文件路径规范化：所有路径经 `resolve()` 规范化后使用 | 参数校验时 |

### 5.2 会话内记忆（短期）

| 属性 | 说明 |
|------|------|
| **存储介质** | Python 内存中的 `messages` 列表（`List[Dict]`） |
| **内容** | 保留用户和助手的所有对话轮次 |
| **用途** | 支持多轮追问和补充信息 |
| **生命周期** | Agent 进程启动时创建，退出时销毁，不写入磁盘 |

**多轮对话示例：**

```
User:  "把 roads.shp 转成 GeoJSON"
Agent: "已识别任务：shp2geojson。输出文件名是什么？"
User:  "输出叫 roads_out.json，放当前目录"
Agent: [生成脚本并请求确认]
```

### 5.3 任务入口手册 Agents.md（长期）

#### 定位

项目级的持久记忆，相当于"给 Agent 的 README"。

#### 存储规范

| 属性 | 值 |
|------|---|
| 文件名 | `Agents.md` |
| 位置 | 工作空间根目录 |
| 格式 | Markdown |
| 编码 | UTF-8 |

#### 加载时机

Agent 启动时，若该文件存在，将其全文注入系统提示词（或作为第一条 `user` 消息）。

#### 内容示例

```markdown
# 项目：城市道路数据处理

- 原始坐标系：EPSG:3857
- 输出坐标系：EPSG:4326
- 常用输入路径：./raw/
- 常用输出路径：./processed/
- 栅格裁剪默认无数据值：0
- 矢量输出格式优先 GeoJSON
```

#### 更新机制

| 更新方式 | 说明 |
|---------|------|
| 手动编辑 | 用户直接修改 `Agents.md` |
| 对话指令 | 通过特定指令（如 `/init 以后都输出4326`）触发更新，**必须经用户确认**后方可写入 |

## 6. 关键设计原则

以下原则为系统设计的最高约束，所有模块实现必须遵守。

| 编号 | 原则名称 | 原则内容 | 违规后果 |
|:----:|---------|---------|---------|
| P1 | **模板化命令，杜绝幻觉** | LLM **绝不直接生成 GDAL 命令**。仅负责选择模板 ID 并填充参数，实际命令由 Jinja2 模板渲染生成。 | 命令错误、数据损坏 |
| P2 | **先展后行** | 任何脚本必须向用户完整展示内容，获得明确确认后才执行。 | 误操作、不可逆数据丢失 |
| P3 | **最小权限** | 默认在工作空间内操作，输出文件默认加时间戳防覆盖；支持外部数据路径。 | 文件被覆盖 |
| P4 | **模板知识唯一** | 用法指导类知识仅来源于 J2 模板元数据，不混合外部网络内容。基础概念类回答由 LLM 参数知识提供。 | 回答不可靠、与系统行动能力脱节 |
| P5 | **依赖极简** | 生产依赖锁定为 `anthropic` 和 `jinja2`。不引入未经批准的第三方库。 | 部署困难、维护成本上升 |

## 7. 验收标准

### 7.1 功能验收清单

| 编号 | 验收项 | 测试输入 | 期望输出/行为 |
|:----:|--------|---------|-------------|
| AC-1 | 文档问答 | "ogr2ogr 能输出哪些格式？" | 能从本地文档中检索并回答几种常见驱动名称 |
| AC-2 | 脚本生成与确认 | "将 `data/roads.shp` 转为 GeoJSON" | 展示如下脚本并等待用户确认：<br>`ogr2ogr -f "GeoJSON" data/roads.json data/roads.shp` |
| AC-3 | 错误处理 | 用不存在的输入文件执行 | 能捕获 GDAL 错误并向用户提示原因 |
| AC-4 | 工作空间隔离 | 指定 `--workspace /tmp/test` | 所有生成文件和执行命令均在该目录下，不影响其他位置 |
| AC-5 | Agents.md 注入 | 工作空间内存在 `Agents.md` | Agent 的首次回答能引用其中设定的默认参数（如自动采用指定坐标系） |

### 7.2 非功能验收标准

| 属性 | 标准 |
|------|------|
| 启动时间 | < 2 秒（无向量数据库加载） |
| 响应延迟 | 简单意图识别 < 2 秒；文档问答 < 5 秒 |
| 依赖数量 | 生产依赖 ≤ 3 个（`anthropic`、`chromadb`、`jinja2`） |

## 8. 后续设计方向

本规格作为后续系统设计的唯一依据。下一阶段的详细设计可围绕以下模块展开：

1. **命令模板结构** — 模板注册表 Schema、参数定义规范、Jinja2 模板语法约定、知识元数据标签规范
2. **模板知识问答策略** — 基础概念/用法指导/错误诊断三层路由、模板匹配逻辑、Prompt 设计
3. **交互状态机** — 用户对话状态流转（空闲 → 意图确认 → 参数收集 → 脚本展示 → 执行）
4. **安全校验层** — 路径规范化实现、命令注入防护、脚本执行机制
